# 7.1 软件供应链安全战略

软件供应链安全战略为保护交付生产环境的组件、供应商、工具和流程提供治理基础。本节建立供应链风险管理所需的战略框架、成熟度模型、组织结构和治理机制。

---

## 7.1.1 供应链攻击图谱

制定防御措施的前提是理解攻击面。供应链威胁可分为六大攻击向量，每类向量具有不同的攻击路径和防御重点。

![供应链攻击向量](../../../assets/images/chapter_07/01_Supply_Chain_Attack_Vectors_v6.png)

### 源代码攻击

源代码攻击指针对源代码仓库或开发者环境的入侵行为。典型攻击手段包括：开发者账户入侵（凭证窃取、多因素认证绕过）、恶意内部提交、仓库劫持（GitHub/GitLab 账户接管）、依赖混淆攻击（将内部包名在公共仓库注册）。

**典型事件**：Codecov Bash Uploader 入侵事件（2021）。攻击者修改 Codecov 的 Bash Uploader 脚本，窃取含敏感信息的环境变量，影响多家企业客户，导致大规模凭证泄露。

**适用边界**：该类防御措施适用于所有使用版本控制系统的开发组织；对于仅使用商业闭源软件、无自研能力的组织，优先级可降低。

**关键约束**：强制 MFA 会增加开发者登录摩擦；GPG 签名提交需要密钥管理基础设施；分支保护规则可能影响紧急修复的响应速度。

**防御措施**：所有开发者强制启用多因素认证；敏感仓库代码审查要求至少 2 名审批者；启用分支保护规则防止强制推送到主分支；使用 GPG 密钥签名提交；通过 CODEOWNERS 文件强制代码所有权。

**常见误区**：（1）认为仅保护主分支即可——攻击者可通过功能分支引入恶意代码后合并；（2）MFA 部署后即认为安全——需定期审计 MFA 覆盖率和绕过路径。

**验证方法**：定期审计开发者账户 MFA 启用率；红队测试模拟账户接管和分支保护绕过；检查 Git 日志中未签名提交的比例。

**运行指标**：MFA 覆盖率（目标：100%）；未签名提交告警触发次数；分支保护规则绕过事件数。

### 构建系统攻击

构建系统攻击指 CI/CD 基础设施被入侵，在构建过程中注入恶意代码。攻击手段包括：构建服务器接管（利用 Jenkins、GitLab CI 等漏洞）、管道配置篡改、从 CI/CD 环境变量窃取密钥、构建后但签名前替换制品。

**典型事件**：SolarWinds Orion 平台攻击（2020）。攻击者获得 SolarWinds 构建系统访问权限，向 Orion 软件更新注入 SUNBURST 后门，受影响客户下载了被入侵的更新。

**适用边界**：任何使用 CI/CD 自动化构建的组织均需关注；手工构建环境的风险模式不同，但并非更安全。

**关键约束**：加固构建环境会增加构建时间和运维复杂度；临时性构建环境需要足够的计算资源；网络隔离可能影响依赖拉取效率。

**防御措施**：加固构建环境（不可变、临时性）；实施构建来源证明（SLSA 框架）；所有制品进行密码学签名；构建网络与生产网络隔离；启用审计日志和异常检测。

**常见误区**：（1）认为托管 CI/CD 服务自动安全——配置不当同样存在风险；（2）仅关注构建产物而忽视构建过程的完整性验证。

**验证方法**：定期对构建环境进行安全评估；检查构建日志中的异常活动模式；验证制品签名的完整性。

**运行指标**：未签名制品告警数；构建环境漏洞扫描发现数；构建过程审计日志完整性。

### 依赖攻击

依赖攻击利用第三方库漏洞或注入恶意包。攻击手段包括：已知漏洞利用（如 Log4Shell）、恶意包注入（拼写劫持、依赖混淆）、维护者账户被入侵（npm、PyPI 接管）、废弃包无安全更新。

**典型事件**：Log4Shell 漏洞（CVE-2021-44228）。Apache Log4j 库中的严重远程代码执行漏洞，广泛部署导致影响范围极大，披露后数小时内即遭利用。

**适用边界**：任何使用开源组件的软件项目均需关注；内部开发的闭源组件同样可能间接依赖受影响的库。

**关键约束**：SCA 工具扫描会产生误报，需要人工分类；依赖版本锁定可能阻碍安全更新的及时应用；私有镜像仓库需要维护成本。

**防御措施**：部署软件成分分析（SCA）工具；实施依赖版本锁定与哈希验证；建立私有包镜像仓库精选组件；CI/CD 中启用自动化漏洞扫描；生成和跟踪 SBOM。

**常见误区**：（1）认为仅扫描直接依赖即可——传递依赖同样是攻击向量；（2）依赖锁定后不再更新——需平衡稳定性与安全更新。

**验证方法**：检查 SCA 工具覆盖率和扫描频率；审计 SBOM 的完整性和时效性；测试依赖混淆攻击的防御有效性。

**运行指标**：已知漏洞依赖数量及严重程度分布；依赖更新平均修复时间；SBOM 覆盖率。

### 分发渠道攻击

分发渠道攻击指软件分发机制被入侵，包括仓库、更新服务器、CDN。攻击手段包括：仓库账户接管（npm、Docker Hub、PyPI）、更新服务器入侵（中间人攻击）、CDN 投毒、镜像站点入侵。

**典型事件**：NPM 包劫持事件。Event-stream npm 包被入侵（2018），攻击者添加依赖以窃取加密货币钱包，被大量下载后才被发现。

**适用边界**：任何从外部源获取软件包或更新的组织均需关注；纯内网环境风险较低但仍需关注镜像源的安全性。

**关键约束**：包签名验证会增加部署复杂度；私有仓库需要存储和带宽成本；多源验证会增加构建时间。

**防御措施**：实施包签名验证；使用具有访问控制的私有仓库；Web 资源使用子资源完整性（SRI）检查；更新通道实施证书固定；建立多源验证机制。

**常见误区**：（1）信任公共仓库的包完整性——公共仓库的安全保障有限；（2）仅验证直接下载的包——缓存和镜像层同样可能被污染。

**验证方法**：测试签名验证机制的有效性；审计仓库访问控制配置；检查 SRI 哈希的覆盖范围。

**运行指标**：签名验证失败告警数；从未授权源拉取包的事件数；仓库访问异常告警数。

### 部署攻击

部署攻击指构建后在部署到生产环境过程中篡改制品。攻击手段包括：仓库中容器镜像替换、Helm Chart 篡改、基础设施即代码（IaC）配置修改、部署管道入侵。

**典型事件**：Docker Hub 恶意镜像事件。流行镜像中隐藏加密货币挖矿程序，具有合法名称的后门基础镜像在大量拉取后才被删除。

**适用边界**：任何使用容器化或 IaC 部署的组织均需关注；传统虚拟机部署的风险模式不同但同样需要制品验证。

**关键约束**：镜像签名和准入控制增加部署延迟；不可变标签策略需要调整现有部署流程；GitOps 工作流对网络连通性有要求。

**防御措施**：实施镜像签名和准入控制（Cosign、Kyverno）；使用不可变标签和来源跟踪；部署仓库扫描和策略执行；遵循最小权限部署凭证原则；采用带审计跟踪的 GitOps 工作流。

**常见误区**：（1）仅扫描镜像而不验证签名——扫描通过不等于来源可信；（2）忽视 IaC 模板的安全审计——配置篡改同样危险。

**验证方法**：测试准入控制策略的有效性；审计部署凭证的权限范围；检查 GitOps 审计日志的完整性。

**运行指标**：准入控制拒绝部署事件数；未签名镜像部署告警数；IaC 配置变更审计覆盖率。

### 运行时攻击

运行时攻击指部署后通过供应链后门进行利用。攻击手段包括：部署后激活的持久化后门、定时炸弹或触发式恶意代码、伪装成合法流量的数据窃取、通过被入侵组件的横向移动。

**典型事件**：SolarWinds SUNBURST 后门运行时行为。后门安装后休眠 2 周，使用域名生成算法（DGA）进行 C2 通信，伪装成合法 Orion 流量。

**适用边界**：任何运行第三方或开源组件的生产环境均需关注；该类攻击检测难度高，需要持续监控能力。

**关键约束**：RASP 会带来性能开销；网络流量分析需要专业能力和工具投入；EDR 部署需要覆盖所有端点。

**防御措施**：部署运行时应用自我保护（RASP）；实施网络流量分析和异常检测；部署终端检测与响应（EDR）；对生产制品进行定期完整性检查；实施零信任网络架构。

**常见误区**：（1）认为通过前期检查即可防止运行时攻击——高级攻击专门设计为通过静态检测；（2）仅关注入站流量——出站异常同样是供应链后门的重要信号。

**验证方法**：红队测试模拟供应链后门激活；检查异常检测规则的覆盖范围；测试完整性检查机制的有效性。

**运行指标**：异常出站连接告警数；完整性校验失败事件数；可疑进程行为告警数。

## 7.1.2 供应链安全成熟度模型

四级成熟度模型为组织持续改进供应链安全能力提供路线图。组织应根据自身现状评估当前等级，制定渐进式提升计划。

### 等级 1：被动响应型（初始级）

**特征**：临时性安全扫描；手动依赖项跟踪；事件驱动的响应；无集中式 SBOM 管理；对第三方风险的可见性有限。

**典型表现**：漏洞响应周期长；团队间存在重复工作；安全标准不一致；SCA 工具误报率高。

**适用场景**：初创企业或刚开始关注供应链安全的组织；安全投入有限、需要快速建立基础能力的团队。

**进阶关键行动**：建立软件资产基线清单；部署基础 SCA 工具；在开发团队中设立安全责任人；记录供应链实践现状。

**约束与权衡**：工具部署初期会产生大量告警需要分类处理；需要获得开发团队的配合以建立资产清单。

### 等级 2：管理规范型（可重复级）

**特征**：CI/CD 管道中的自动化 SCA；集中式策略执行；关键应用的 SBOM 生成；基础供应商安全评估；明确的角色和职责。

**能力表现**：每次提交时进行漏洞扫描；基于策略的组件审批工作流；严重 CVE 的自动化告警；定期供应商安全审查；SLSA 等级 1 来源证明。

**适用场景**：中型企业或有一定安全成熟度的组织；需要满足基础合规要求的团队。

**进阶关键行动**：将 SBOM 生成集成到所有构建管道；实施制品签名（Cosign、Sigstore）；建立供应链安全治理委员会；部署依赖项持续监控。

**约束与权衡**：制品签名需要密钥管理基础设施；策略执行可能阻断构建流程需要例外处理机制。

### 等级 3：主动防御型（已定义级）

**特征**：持续漏洞监控；SLSA 等级 2 以上合规；自动化 SBOM 增强和分析；实时威胁情报集成；零信任供应链架构。

**能力表现**：基于风险的漏洞优先级排序；自动化补丁工作流；构建环境加固（临时性、隔离）；全面供应商风险评分；供应链威胁狩猎。

**适用场景**：大型企业或高安全要求行业（金融、医疗）；面临监管合规压力的组织。

**进阶关键行动**：关键系统达到 SLSA 等级 3；实施供应链威胁情报平台；自动化漏洞响应工作流；定期开展供应链攻击模拟演练。

**约束与权衡**：高级威胁情报平台投入较大；SLSA 等级 3 的临时性构建环境对基础设施要求高。

### 等级 4：持续优化型（管理与度量级）

**特征**：完全供应链透明度；基于行为分析的异常检测；主动威胁狩猎；参与行业威胁情报共享。

**能力表现**：SLSA 等级 4 密封构建；自动化供应链风险量化；实时 SBOM 差异对比和影响分析。

**适用场景**：关键基础设施运营商；需要达到行业领先安全水平的组织。

**维持活动**：与同行持续基准对比；新技术研发；公开披露和威胁情报共享；对供应商生态施加安全影响力。

**约束与权衡**：需要持续的研发投入；密封构建对开发灵活性有一定影响；威胁情报共享需要平衡信息披露风险。

### 成熟度评估模板

以下评分卡用于评估当前成熟度水平。评估时应关注各能力域的最薄弱环节，而非平均水平。

| 能力域 | 等级 1 | 等级 2 | 等级 3 | 等级 4 |
|-------|--------|--------|--------|--------|
| **SBOM 覆盖率** | 低覆盖 | 部分覆盖 | 高覆盖 | 全覆盖 |
| **SCA 自动化** | 手动 | CI/CD 集成 | 持续监控 | 智能驱动 |
| **签名与验证** | 无 | 基础签名 | SLSA L2+ | SLSA L4 |
| **供应商管理** | 临时性 | 定期审查 | 持续监控 | 预测性风险 |
| **事件响应** | 被动响应 | 流程定义 | 自动化 | 主动狩猎 |
| **威胁情报** | 无 | 基础订阅 | 集成应用 | 预测分析 |

**评估说明**：大多数能力域落在哪个等级，即为当前整体等级。改进工作应优先聚焦得分最低的领域，因为安全能力遵循木桶效应。

## 7.1.3 组织架构与角色

### 组织模型

供应链安全的有效治理需要跨职能协作，而非建立独立的专项团队。推荐采用 GRC 主导的安全风险管理委员会统筹模型。

```
        安全风险管理委员会
              [GRC主导]
                 |
    +-----+-----+-----+-----+-----+
    |     |     |     |     |     |
  AppSec TPRM  SOC  Cloud  Data
  [供应链] [供应商] [威胁] [云安全] [数据]
```

**设计原则**：所有风险域（供应链、数据、云、应用等）的安全治理统一由 GRC 主导的安全风险管理委员会统筹，不设立独立的供应链安全办公室、数据安全委员会、云安全委员会等，避免组织碎片化和职责重叠。

### 核心角色与职责

#### GRC 团队（供应链安全职能）

**汇报路线**：CISO

**职责范围**：制定供应链安全战略和政策；统筹安全风险管理委员会供应链议题；维护 SBOM 仓库和漏洞数据库；监督供应商安全评估计划；协调供应链入侵事件响应；推动 NIST SSDF、SLSA、监管要求合规；发布安全公告和指南；管理供应链安全工具和平台。

**关键角色**：供应链安全负责人（战略、治理、高管汇报）；SBOM 架构师（标准、工具、自动化）；漏洞管理员（CVE 跟踪、优先级排序、补丁管理）；威胁情报分析师（供应链威胁监控）；合规专家（监管映射、审计支持）。

#### 应用安全团队（AppSec）

**接触点**：SBOM 生成、安全 CI/CD

**职责范围**：将 SCA 工具集成到开发流水线；为所有应用生成并发布 SBOM；实施制品签名和验证；培训开发者安全依赖管理；在 CI/CD 门禁强制执行安全策略。

**关键交付物**：每个生产版本的 SBOM；带来源证明的签名制品；安全 CI/CD 流水线模板；开发者安全培训材料。

#### 第三方风险管理（TPRM）

**接触点**：供应商和服务商安全

**职责范围**：评估供应商安全态势（问卷、审计）；监控供应商安全记分卡；管理包含安全 SLA 的供应商合同；跟踪第四方风险（供应商的供应商）；协调供应商事件响应。

**关键交付物**：供应商风险评级和批准决策；持续监控仪表板；供应商安全要求文档；事件沟通协议。

#### 安全运营中心（SOC）

**接触点**：威胁检测和事件响应

**职责范围**：监控供应链威胁情报源；检测构建过程和部署中的异常；对供应链安全告警进行分类；执行事件响应手册；对供应链入侵进行取证分析。

**关键交付物**：全天候供应链威胁监控；事件响应手册；威胁情报报告；事后分析报告。

#### 云安全团队

**接触点**：容器和基础设施供应链

**职责范围**：加固容器镜像供应链；管理私有注册表和镜像扫描；实施准入控制策略（OPA、Kyverno）；保护 IaC 流水线（Terraform、CloudFormation）；监控云市场安全。

**关键交付物**：加固的基础镜像；镜像扫描策略；注册表访问控制；IaC 安全模板。

### RACI 矩阵

下表定义了关键活动的职责分配，确保问责清晰、协作顺畅。

| 活动 | GRC | AppSec | TPRM | SOC | Cloud |
|-----|-----|--------|------|-----|-------|
| 制定供应链战略 | R/A | C | C | C | C |
| 生成 SBOM | C | R/A | I | I | C |
| 供应商安全评估 | A | I | R | I | I |
| 威胁情报监控 | C | I | I | R/A | C |
| 容器镜像扫描 | C | C | I | C | R/A |
| 合规报告 | R/A | I | C | I | I |
| 事件响应（供应链） | A | C | C | R | C |

**说明**：R = 负责执行；A = 最终问责；C = 咨询参与；I = 知情通知

## 7.1.4 框架对齐：NIST SSDF 与 SLSA

### NIST 安全软件开发框架（SSDF）

NIST SP 800-218 定义了四大实践组，为软件供应链安全提供系统化指导。

![SLSA 框架](../../../assets/images/chapter_07/03_SLSA_Framework_v6.png)

#### PO：组织准备（Prepare the Organization）

| 实践 | 供应链应用 |
|-----|-----------|
| PO.1 定义安全要求 | 在要求中包含供应链安全（SBOM、签名） |
| PO.2 实施角色和职责 | 建立 GRC 主导的跨职能协作 |
| PO.3 实施支持工具链 | 部署 SCA、SBOM、签名工具 |
| PO.4 定义安全开发标准 | 安全编码标准中的供应链安全 |
| PO.5 实施安全开发环境 | 加固的 CI/CD、受保护的源代码仓库 |

#### PS：保护软件（Protect the Software）

| 实践 | 供应链应用 |
|-----|-----------|
| PS.1 保护代码免受未授权访问 | 分支保护、MFA、代码签名 |
| PS.2 提供验证机制 | 制品签名、SLSA 来源证明 |
| PS.3 保护制品 | 注册表访问控制、签名验证 |

#### PW：生产高安全性软件（Produce Well-Secured Software）

| 实践 | 供应链应用 |
|-----|-----------|
| PW.1 安全设计软件 | 威胁建模包含供应链风险 |
| PW.2 审查代码 | 在代码审查中包含依赖变更 |
| PW.4 复用现有软件 | 批准的依赖源、SCA 扫描 |
| PW.5 创建 SBOM | 为每个版本生成 SBOM |
| PW.6 安全配置软件 | 安全默认值、无硬编码密钥 |
| PW.7 审查和测试代码 | 在测试中包含 SCA、SBOM 验证 |
| PW.8 安全配置构建工具 | 不可变构建环境、审计日志 |
| PW.9 构建来源证明 | SLSA 来源证明 |

#### RV：响应漏洞（Respond to Vulnerabilities）

| 实践 | 供应链应用 |
|-----|-----------|
| RV.1 识别漏洞 | 持续 SCA 扫描、威胁情报 |
| RV.2 评估、优先级排序、修复 | 基于 SBOM 的影响分析、补丁管理 |
| RV.3 分析响应 | 事后审查、经验教训 |

### SLSA（Supply-chain Levels for Software Artifacts）

SLSA 提供渐进式安全保障等级，组织可根据系统关键程度选择目标等级。

#### SLSA 等级 1：来源证明存在

**要求**：构建过程生成来源证明；来源证明包含构建指令和依赖信息。

**实施方式**：使用 GitHub Actions attestations；生成包含构建信息的基础 SBOM。

**价值**：基础可追溯性，审计跟踪。

**约束**：来源证明可能被伪造，仅提供基础保障。

#### SLSA 等级 2：签名来源证明 + 托管构建

**要求**：来源证明经过密码学签名；由构建服务（非开发者）生成来源证明；来源证明具有防篡改能力。

**实施方式**：集成 Sigstore/Cosign 进行签名；使用托管 CI/CD（GitHub Actions、GitLab CI）；在不可变日志（Rekor）中存储来源证明。

**价值**：防止来源证明篡改，可验证构建。

**约束**：需要签名基础设施；托管服务可能不满足数据主权要求。

#### SLSA 等级 3：加固构建 + 不可伪造来源证明

**要求**：隔离的构建环境（临时性）；记录源代码和构建平台元数据；来源证明即使构建服务管理员也无法伪造。

**实施方式**：临时性构建运行器（Docker、Firecracker）；密封构建（所有依赖声明）；构建组件间双向 TLS。

**价值**：强完整性保障，防止内部威胁。

**约束**：临时性环境增加构建时间；密封构建需要完整的依赖声明。

#### SLSA 等级 4：双人审查 + 密封构建

**要求**：所有变更经两名可信人员审查；密封、可重现构建；所有依赖完全声明。

**实施方式**：强制代码审查（分支保护）；使用 Bazel 或 Nix 进行密封构建；可重现构建验证。

**价值**：最高保障等级，适用于关键基础设施。

**约束**：双人审查增加发布延迟；可重现构建对工具链要求高。

**目标等级建议**：内部工具——SLSA 等级 1；生产服务——SLSA 等级 2；金融/PII 系统——SLSA 等级 3；关键基础设施——SLSA 等级 4。

## 7.1.5 治理机制

### 安全风险管理委员会（供应链专题）

**组成**：主席由 GRC 负责人或 CISO 担任；成员包括 AppSec、TPRM、SOC、Cloud Security、Data Security、Engineering、Legal 代表。

**说明**：供应链安全议题由统一的安全风险管理委员会处理，不单独设立供应链安全委员会。

**频率**：月度例会 + 重大事件临时会议。

**职责**：审查和批准供应链安全政策；确定漏洞修复优先级；审批供应商安全豁免/例外；分配工具和项目预算；向高管层升级关键供应链风险；审查指标和成熟度进展。

**输出**：政策审批和更新；风险接受决策；预算分配；高管风险报告。

### 政策框架

**层级结构**：

```
供应链安全政策（执行层）
         ↓
    标准（技术要求）
         ↓
    程序（分步流程）
         ↓
    指南（实践参考）
```

#### 核心政策

**组件审批政策**：所有第三方组件使用前必须获得审批；审批基于安全评估（CVE 历史、维护者声誉、许可证）；高风险组件需要额外审查；禁止使用已废弃或无人维护的组件。

**SBOM 生成政策**：所有生产软件发布需要 SBOM；SBOM 必须满足 NTIA 最低要素要求；SBOM 存储在集中式仓库；向客户交付软件时包含 SBOM。

**制品签名政策**：所有生产制品必须经过密码学签名；签名密钥在 HSM 或云 KMS 中管理；部署前强制签名验证；要求来源证明（SLSA 等级 2 以上）。

**供应商安全政策**：基于数据访问和关键程度进行供应商风险分级；入职前需要安全评估；关键供应商需要持续监控；合同中包含事件披露 SLA。

**漏洞管理政策**：严重漏洞需在规定时限内修复；高危漏洞需在规定时限内修复；如延迟修复需要补偿性控制措施；强制基于 SBOM 的影响分析。

### 指标与报告

**关键绩效指标（KPI）**：

| 指标 | 度量频率 |
|-----|---------|
| SBOM 覆盖率（生产应用占比） | 月度 |
| 制品签名合规率 | 周度 |
| 严重漏洞平均修复时间（MTTP） | 周度 |
| 供应商评估覆盖率 | 季度 |
| SLSA 等级 2 以上采用率（关键系统） | 季度 |
| SCA 告警误报率 | 月度 |

**汇报节奏**：周度——漏洞仪表板给工程团队负责人；月度——指标和趋势分析给安全风险管理委员会；季度——执行摘要给 CISO 和 CTO；年度——董事会汇报，包含风险趋势和战略计划。

## 7.1.6 预算与资源规划

### 预算分配

供应链安全投入通常占整体安全预算的一定比例，具体取决于组织的风险敞口和成熟度目标。

**预算分解**：

| 类别 | 说明 |
|-----|------|
| 工具和平台 | SCA 工具、SBOM 平台、签名基础设施、威胁情报 |
| 人员 | GRC 团队供应链安全职能、承包商支持、培训 |
| 供应商评估 | 第三方审计、安全问卷、渗透测试 |
| 合规与审计 | 监管合规、认证计划 |
| 研发与创新 | 概念验证、新兴威胁研究 |

### 投资合理性论证

**成本规避**：供应链入侵可导致直接财务损失、监管罚款和客户流失；主动投资供应链安全可降低这些风险发生的概率和影响程度。

**效率提升**：自动化漏洞扫描减少人工工作量；基于 SBOM 的影响分析加速事件响应；集中式 SCA 减少工具分散成本。

**竞争优势**：安全作为销售差异化要素（满足客户采购要求）；加速合规认证（SOC 2、ISO 27001）；可能降低网络保险费率。
