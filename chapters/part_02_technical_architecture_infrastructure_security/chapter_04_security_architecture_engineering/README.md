# Chapter 4: Security Architecture and Engineering | 安全架构与工程

## 构建可扩展、可验证的企业安全架构体系

> **核心定位**：为全球化企业提供从战略到技术实现的 security architecture framework、设计原则、威胁建模方法与参考架构，支撑业务安全创新与风险管控。

---

## 本章导读

### 学习目标

完成本章学习后，读者将能够：

1. 掌握架构框架：理解并应用 SABSA、TOGAF、Zachman、零信任等主流 security architecture framework
2. 应用设计原则：运用最小权限、纵深防御、职责分离等核心原则指导架构决策
3. 实施威胁建模：使用 STRIDE、PASTA、Attack Trees 等方法进行系统化威胁识别与风险评估
4. 设计架构模式：构建零信任、多层防御、微隔离、SSE/SASE 等现代安全架构
5. 执行架构评审：建立标准化的架构评审流程、安全设计评审机制与控制验证方法
6. 优化技术选型：制定技术选型框架，平衡自建与购买、开源与商业等决策
7. 应用参考架构：在网络、云、应用、数据等领域落地安全参考架构
8. 管理架构债务：识别技术债务、规划架构重构与现代化路径

### 适用场景

本章内容适用于以下场景：

- 架构转型企业：从传统安全架构向零信任、云原生架构转型
- 全球化企业：需要跨区域、跨云平台的统一安全架构体系
- 架构规范化：建立企业级安全架构治理、评审与标准化机制
- 风险驱动设计：通过威胁建模将风险前置到设计阶段
- 云原生迁移：构建适配多云、混合云的安全参考架构
- 遗留系统现代化：管理技术债务、规划架构演进路径
- 工程能力提升：建立 architecture-as-code、自动化控制验证等工程实践

| ---------- | ------------------------------------------------------------------------ | ------------------------------------------------------------------------------------ |

| TOGAF      | The Open Group Architecture Framework                                    | 企业架构框架，包含架构开发方法（ADM）与安全架构集成                                  |

| DREAD      | Damage, Reproducibility, Exploitability, Affected Users, Discoverability | 风险评级方法（已部分废弃，但仍有参考价值）                                           |
| SDR        | Security Design Review                                                   | 安全设计评审，在设计阶段评估架构安全性                                               |
| ARB        | Architecture Review Board                                                | 架构评审委员会，负责架构决策与例外批准                                               |
| SSE        | Security Service Edge                                                    | 安全服务边缘，整合 CASB、SWG、ZTNA、FWaaS 的云安全架构                               |
| SASE       | Secure Access Service Edge                                               | 安全访问服务边缘，SSE + SD-WAN 的融合架构                                            |
| AaaC       | Architecture-as-Code                                                     | 架构即代码，通过代码管理架构资产与控制策略                                           |
| IaC        | Infrastructure-as-Code                                                   | 基础设施即代码，通过代码定义基础设施配置                                             |
| PaC        | Policy-as-Code                                                           | 策略即代码，通过代码定义安全策略与合规规则                                           |
| TCO        | Total Cost of Ownership                                                  | 总拥有成本，包含采购、实施、运营、维护的全生命周期成本                               |
| POC        | Proof of Concept                                                         | 概念验证，小规模验证技术方案的可行性                                                 |

## 章节结构

### Part 1: 架构基础（Architecture Foundation）

**4.1 安全架构框架（Security Architecture Frameworks）** 

SABSA 框架（五层架构模型：业务、架构师、设计师、构建者、设施管理员）；TOGAF 安全架构（ADM 方法、安全架构集成）；Zachman 框架（6×6 矩阵、制品分类）；企业安全架构（ESA 多层视图）；零信任架构框架（NIST SP 800-207）。

**4.2 架构设计原则（Architecture Design Principles）** 

最小权限原则（least privilege）；纵深防御（defense in depth）；职责分离（separation of duties）；公开设计（open design，Kerckhoffs's principle）；经济机制（economy of mechanism，simplicity）；安全默认（secure by default，fail-safe defaults）。

### Part 2: 威胁与风险（Threat & Risk）

**4.3 威胁建模（Threat Modeling）** 

STRIDE 方法（欺骗、篡改、否认、信息泄露、拒绝服务、特权提升）；PASTA 方法（七阶段威胁分析流程）；attack trees（攻击树建模与路径分析）；DREAD 风险评级（损害、可重现性、可利用性、影响用户、可发现性）；威胁建模工具（Microsoft Threat Modeling Tool、IriusRisk、Threagile）。

### Part 3: 架构设计（Architecture Design）

**4.4 安全架构模式（Security Architecture Patterns）** 

零信任架构（zero trust：身份、设备、网络、应用、数据）；多层防御架构（defense in depth：边界、网络、主机、应用、数据）；微隔离架构（microsegmentation：网络分段、工作负载隔离）；安全服务边缘（SSE/SASE：CASB、SWG、ZTNA、FWaaS、SD-WAN）；身份中心架构（identity-centric：统一身份、策略决策、持续评估）。

**4.5 架构评审与验证（Architecture Review and Validation）** 

安全架构评审流程（申请、初审、正式评审、跟踪、关闭）；安全设计评审（SDR：评审检查表、风险矩阵、决策记录）；架构威胁分析（集成威胁建模、攻击面分析）；安全控制验证（控制映射、自动化测试、符合性检查）；架构文档规范（视图标准、工件模板、版本管理）。

### Part 4: 技术决策（Technology Decision）

**4.6 技术选型与决策（Technology Selection and Decision Making）** 

技术选型框架（需求分析、市场调研、评分矩阵、决策流程）；自建与购买（build vs. buy：成本、时间、能力、风险评估）；开源与商业（许可证、支持、社区、安全性、合规性）；POC 与技术评估（验证计划、测试场景、评估标准、决策建议）；TCO 与 ROI 分析（采购、实施、运营、维护成本；价值量化）。

**4.7 安全参考架构（Security Reference Architectures）** 

网络安全参考架构（分段、区域、边界、零信任网络）；云安全参考架构（landing zone、多账户、工作负载保护、CSPM）；应用安全参考架构（SDLC 集成、API 安全、微服务安全）；数据安全参考架构（分类分级、加密、脱敏、访问控制、DSPM）；零信任参考架构（端到端零信任：身份、设备、网络、应用、数据）。

### Part 5: 持续演进（Continuous Evolution）

**4.8 架构演进与债务管理（Architecture Evolution and Technical Debt）** 

技术债务识别与量化（债务分类、影响评估、优先级排序）；架构重构策略（渐进式重构、绞杀者模式、大爆炸重构）；遗留系统安全加固（风险评估、补偿控制、隔离策略）；架构现代化路径（云原生改造、微服务拆分、零信任迁移）；架构迁移风险管理（风险识别、缓解措施、回滚计划）。

**4.9 实战案例（Case Studies）** 

案例 1：零信任架构设计与实施；案例 2：SABSA 框架落地；案例 3：威胁建模实战；案例 4：技术选型决策。

## 常见问题

**Q1：SABSA 和 TOGAF 如何选择？**

两者可以结合使用。SABSA 专注于安全架构，业务驱动，五层模型清晰，适合需要将安全投资与业务成果直接关联的场景。TOGAF 是通用企业架构框架，ADM 方法成熟，适合已有企业架构团队的大型企业。推荐方案是使用 TOGAF ADM 作为整体架构方法，在安全域使用 SABSA 细化。

**Q2：威胁建模何时进行？**

建议在以下时机进行：新产品或功能设计阶段（需求确定后、开发前）；架构重大变更前（如云迁移、零信任改造）；集成第三方服务时；发生重大安全事件后（复盘与改进）；定期复审（建议每年或每个大版本）。

**Q3：如何衡量架构工作的价值？**

使用多维度指标：预防价值（设计阶段发现的缺陷数量与严重程度）；效率提升（模式复用减少的重复工作量）；风险降低（架构评审避免的潜在损失）；合规保障（通过审计的控制点覆盖率）；业务支撑（架构评审周期对业务上线的影响）。

**Q4：技术债务如何优先级排序？**

使用风险与影响矩阵：高风险 + 高影响（如暴露互联网的遗留系统）应立即处理；高风险 + 低影响（如非关键系统的弱加密）应计划处理；低风险 + 高影响（如性能问题但无安全风险）应监控并计划；低风险 + 低影响应记录在债务登记册后接受或延期。

**Q5：零信任架构从哪里开始？**

推荐渐进式实施路径：阶段 1（0-6 个月）统一身份、MFA、条件访问；阶段 2（6-12 个月）部署 ZTNA、微隔离关键应用；阶段 3（12-18 个月）应用层零信任、API 安全；阶段 4（18-24 个月）数据层零信任、持续信任评估。

## 常见陷阱

| 陷阱           | 描述                             | 避免方法                                    |
| -------------- | -------------------------------- | ------------------------------------------- |
| 过度设计       | 追求完美架构，忽视交付时间与成本 | 使用 MVP 方法，渐进式改进；设定"够用"的标准 |
| 框架僵化       | 照搬框架而不结合企业实际         | 将框架作为参考而非教条；根据成熟度裁剪      |
| 评审流于形式   | 评审走过场，缺乏实质审查         | 使用结构化检查表；要求输出威胁模型、ADR     |
| 忽视技术债务   | 只关注新架构，忽视遗留系统       | 建立债务登记册；定期复审；设定关闭目标      |
| 缺乏度量       | 无法证明架构工作的价值           | 建立架构度量看板；量化预防价值与效率提升    |
| 安全与业务脱节 | 架构决策不考虑业务影响           | 与 BISO 协同（Chapter 3）；业务风险评估     |
| 文档滞后       | 架构文档与实现不一致             | 使用 Architecture-as-Code；自动化文档生成   |
| 工具过载       | 引入过多工具，增加复杂度         | 技术选型时考虑整合性；优先平台化方案        |

## 关键检查清单

架构评审检查清单（checklist）

业务背景：

- 业务目标与安全需求明确
- 利益相关方已识别
- 合规要求已分析

威胁建模：

- 已完成威胁建模（STRIDE/PASTA）
- 高风险威胁已识别并缓解
- 攻击面已分析并最小化

架构设计：

- 符合企业架构原则（最小权限、纵深防御等）
- 使用标准安全模式（模式库）
- 架构图完整（逻辑、物理、数据流）

控制验证：

- 安全控制已映射到威胁
- 控制可验证（优先自动化）
- 监控与响应机制已定义

技术选型：

- 技术选型有决策记录（ADR）
- 已评估 TCO 与 ROI
- 供应商安全性已评估（TPRM）

合规性：

- 满足适用的法规要求（GDPR、PIPL 等）
- 满足行业标准（PCI DSS、ISO 27001 等）
- 满足内部安全政策

可运营性：

- 运维流程已定义
- 监控与告警已配置
- 事件响应计划已制定
- 灾难恢复与业务连续性已规划

文档与交接：

- 架构文档完整（符合模板规范）
- 运维文档已准备
- 知识转移已完成
