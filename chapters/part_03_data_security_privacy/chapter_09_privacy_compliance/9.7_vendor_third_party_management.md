# 9.7 供应商与第三方管理

## 概述

当企业将个人数据的处理活动委托给外部供应商时，数据控制者的合规责任并未转移。GDPR 第 28 条明确规定，数据控制者必须"仅使用提供充分保障的处理者"，并通过书面合同（数据处理协议，DPA）约束处理者的行为。这一要求构成供应商隐私治理的法律基础。

本节阐述供应商隐私管理的关键控制点：DPA 必备条款、供应商风险评估方法、子处理者管理流程、国际数据传输机制选择，以及供应商数据泄露的响应协调。

---

## 9.7.1 数据处理协议（DPA）要素

### GDPR 第 28 条核心要求

数据处理协议是控制者与处理者之间的强制性书面合同。该协议须明确以下要素：处理的主题事项与期限、处理的性质与目的、个人数据类型、数据主体类别、控制者与处理者的权利义务。

处理者的核心义务可归纳为八项（对应 GDPR 第 28 条第 3 款 a-h 项）：

1. 指示约束：仅按控制者书面指示处理数据，除非法律另有要求
2. 保密义务：确保获授权处理数据的人员已签署保密协议
3. 安全措施：实施 GDPR 第 32 条规定的技术和组织措施
4. 子处理者限制：未经控制者书面授权不得启用子处理者
5. 协助响应权利：协助控制者响应数据主体权利请求
6. 合规协助：协助控制者履行 DPIA、监管机构咨询等义务
7. 终止处理：合同终止后删除或返还所有个人数据
8. 审计配合：提供审计所需信息，允许控制者或指定审计方进行检查

### DPA 必备条款检查清单

以下条款的完整性直接影响 DPA 的合规有效性。选择这些维度是因为它们直接对应 GDPR 第 28 条的强制要求，缺失任一条款都可能导致 DPA 无法满足法定标准。

| 条款类别   | 必备内容                         | 验证要点                   |
| ---------- | -------------------------------- | -------------------------- |
| 主题事项   | 服务描述、期限、处理性质与目的   | 是否与实际服务范围一致     |
| 数据范围   | 个人数据类型、数据主体类别       | 是否涵盖所有实际处理的数据 |
| 处理者义务 | 上述八项义务的具体约定           | 每项义务是否有明确条款     |
| 技术措施   | 加密标准、访问控制、事件响应时限 | 是否可量化验证             |
| 子处理者   | 当前列表、新增通知机制、反对权   | 通知时限与反对程序是否明确 |
| 国际传输   | 传输机制（SCC/BCR/DPF）、目的国  | 是否符合 Schrems II 要求   |
| 审计权利   | 频率、通知期限、范围、替代机制   | 是否具有可执行性           |
| 泄露通知   | 时限、必须包含的信息项           | 是否满足监管通知窗口要求   |
| 终止处理   | 数据返还/删除方式、验证机制      | 删除完成的证明要求         |

上述维度构成 DPA 合规性的最低要求。实践中，控制者还可根据自身风险偏好增加附加条款（如网络安全责任险要求、渗透测试报告提供义务等），但上表所列为法定底线。

适用边界：DPA 适用于所有代表控制者处理个人数据的供应商，包括云基础设施、SaaS 平台、外包服务商、数据分析服务商等。若供应商仅提供不涉及个人数据访问的服务（如纯硬件采购），则无需签署 DPA。

常见误区：

- 将服务协议（MSA）等同于 DPA，未检查是否包含 GDPR 第 28 条所有必备条款
- DPA 签署后不再更新，忽略服务范围变更或数据类型新增带来的协议修订需求
- 接受供应商的标准 DPA 模板而未审查条款是否满足本企业的特定合规要求

验证方法：建立 DPA 合规检查清单，逐项核对上述必备条款的存在性与完整性。对于关键供应商，建议由法务或隐私团队进行二次审查。

## 9.7.2 供应商隐私风险评估

### 评估框架设计

供应商隐私风险评估需从三个维度展开：数据访问风险、供应商成熟度、地理与监管风险。评估结果决定后续的管控措施强度。

数据访问风险评估供应商接触数据的敏感程度与范围。评估因素包括：处理的数据量级、是否涉及特殊类别数据（健康、生物识别、宗教信仰等）、访问权限级别（只读/读写/完全访问）、是否存储数据副本。

![供应商风险管理](../../../assets/images/chapter_07/06_Vendor_Risk_Management_v6.png)

供应商成熟度评估其隐私保护能力。正向指标包括：持有 ISO 27701、ISO 27001、SOC 2 Type II 等认证、已任命数据保护官（DPO）、隐私政策透明且易于获取。负向指标包括：过去一定期限内发生过数据泄露事件、缺少独立的隐私管理职能。

地理与监管风险评估数据存储与处理的司法管辖区风险。需关注：数据是否存储在政府监控风险较高的司法区域、是否涉及跨境传输、供应商是否受域外法律（如美国 CLOUD Act）约束。

### 风险分级与控制措施

基于综合评估结果，将供应商划分为不同风险等级，并对应差异化的管控要求：

高风险供应商（如处理大量敏感数据、数据存储在高风险司法区域）：

- 审批要求：CISO 或 DPO 审批
- 评估要求：完整 DPIA 评估
- 审计频率：年度现场审计
- 合同强化：考虑数据本地化条款
- 复审周期：每季度复审

中风险供应商（如处理中等量级普通个人数据）：

- 审批要求：隐私团队审批
- 评估要求：简化 DPIA
- 审计频率：年度问卷审查或远程审计
- 合同要求：标准 DPA
- 复审周期：半年度复审

低风险供应商（如仅访问有限的非敏感数据）：

- 审批要求：采购团队审批
- 评估要求：DPA 签署
- 审计频率：接受第三方认证（如 SOC 2）替代
- 复审周期：年度复审

关键约束：

- 风险评估需在供应商入围前完成，避免"先用后补"的被动局面
- 评估标准需与业务部门对齐，过严的标准可能阻碍正常业务运营
- 地理风险评估需持续更新，以反映监管环境变化（如充分性决定的生效或撤销）

验证方法：

- 抽查新供应商入围记录，验证风险评估是否在合同签署前完成
- 对比供应商自报信息与公开可查信息（如认证状态、泄露历史）的一致性
- 定期复审供应商清单，验证风险等级是否随业务变化而更新

运行指标：

- 供应商评估完成率：新供应商入围前完成评估的比例
- 高风险供应商占比：监控供应商组合的整体风险分布
- 评估到期未复审数：超过复审周期未更新评估的供应商数量

## 9.7.3 子处理者管理

### 子处理者授权与通知机制

GDPR 第 28 条第 2 款规定，处理者启用子处理者须获得控制者的书面授权。授权方式分为两种：特定授权（逐一批准）与一般授权（预先同意，但须通知变更并保留反对权）。

实践中，大多数 SaaS 供应商采用一般授权模式，其核心控制点包括：

1. 当前子处理者清单公开：供应商须在网站或协议附录中列明当前所有子处理者的名称、服务内容、数据处理位置
2. 新增通知机制：启用新子处理者前，提前一定天数通知控制者（常见约定为提前 30 天）
3. 反对权保障：控制者在通知期内有权反对新子处理者的任命；若反对，供应商应提供替代方案或允许合同终止
4. 责任流转：处理者与子处理者之间须签署"背靠背"协议，将主 DPA 的所有义务流转至子处理者

### 子处理者生命周期管理

企业作为控制者，应建立子处理者变更的监控机制：

- 订阅通知：订阅主要供应商的子处理者变更通知（多数供应商提供邮件订阅或 RSS）
- 变更评估：收到通知后，评估新子处理者是否引入额外风险（如数据传输至新的司法管辖区）
- 行使反对权：若评估结果不可接受，在约定期限内书面反对，并启动替代方案讨论
- 记录留存：保留所有通知与评估决策记录，以备审计

常见误区：

- 签署 DPA 后不再关注子处理者变更，导致数据流向不可控
- 仅关注一级子处理者，忽略子处理者的子处理者（"子处理者链"）风险
- 反对权流于形式，未建立评估与响应流程

验证方法：

- 定期核对供应商公开的子处理者列表与最近一次通知记录
- 抽查子处理者变更评估记录，验证是否在约定期限内完成
- 审计子处理者协议签署情况，确认责任流转条款完整

## 9.7.4 供应商审计与持续监控

### 基于风险的审计策略

审计频率与深度应与供应商风险等级匹配。以下对比展示不同风险等级对应的审计要求差异，帮助组织合理配置审计资源。

| 风险等级 | 审计类型              | 中期检查     | 认证监控     | 泄露演练   |
| -------- | --------------------- | ------------ | ------------ | ---------- |
| 高风险   | 年度现场审计          | 季度问卷     | 持续监控     | 半年度测试 |
| 中风险   | 两年一次现场/远程审计 | 年度问卷     | 年度核查     | 年度测试   |
| 低风险   | 接受 SOC 2 报告替代   | 两年一次问卷 | 两年一次核查 | 合同测试   |

该策略的核心权衡在于：高风险供应商需要深度监督但审计成本较高，低风险供应商可通过认证报告替代以降低运营负担。组织应根据自身资源与风险偏好调整具体频率。

### 审计检查域

供应商隐私审计应覆盖以下检查域：

合同合规：DPA 条款执行情况、子处理者变更是否按约定通知、数据存储位置是否符合约定、数据保留期限是否遵守

技术措施：加密实施状态（传输与静态）、访问控制机制（最小权限与多因素认证）、日志记录与监控能力、漏洞管理与补丁管理流程

组织措施：隐私政策与程序文档、员工培训记录、敏感岗位背景调查、保密协议签署覆盖率

事件响应：事件响应计划与团队、通知时效机制测试（验证是否能在约定时限内通知控制者）、历史事件复盘记录

DSR 协助能力：协助控制者响应数据主体请求的流程、响应时效能力、数据导出格式与删除验证机制

### 持续监控机制

除定期审计外，应建立持续监控能力：

- 认证到期监控：跟踪供应商关键认证（ISO 27001、SOC 2）的有效期，设置提前提醒
- 公开信息监测：监控供应商相关的安全事件新闻、监管处罚信息
- SLA 违反告警：监控供应商服务水平协议的执行情况，特别是安全相关条款
- 数据量异常检测：对与供应商之间的数据传输量建立基线，检测异常波动

关键约束：

- 审计成本与收益需平衡，低风险供应商可接受认证报告替代现场审计
- 审计发现需有闭环跟踪机制，否则审计流于形式
- 持续监控需适度，避免过度干预供应商正常运营

验证方法：

- 审计发现的整改完成率与平均整改周期
- 年度审计计划执行率
- 认证到期提醒的及时性

运行指标：

- 供应商审计覆盖率：按风险等级统计的审计执行比例
- 审计发现整改率：已关闭发现项占总发现项的比例
- 认证过期供应商数：持有过期认证仍在服务的供应商数量

## 9.7.5 国际数据传输机制

### 传输机制选择

当个人数据从欧盟/欧洲经济区传输至第三国时，须选择合适的传输机制。可选机制按优先级排列如下：

充分性决定：若目的国已获欧盟委员会充分性认定（如英国、日本、韩国、加拿大等），则无需额外保障措施即可传输。这是最简便的传输机制。

EU-US 数据隐私框架（DPF）：2023 年 7 月生效，适用于向美国 DPF 认证组织的传输。该框架替代了被 Schrems II 判决废止的 Privacy Shield。控制者需核实美国接收方是否在 DPF 清单中。

标准合同条款（SCC）：欧盟委员会批准的标准合同，适用于无充分性决定的第三国传输。2021 版 SCC 包含四个模块，需根据传输双方角色选择：

- 模块 1：控制者到控制者
- 模块 2：控制者到处理者（最常见）
- 模块 3：处理者到处理者
- 模块 4：处理者到控制者

约束性公司规则（BCR）：适用于跨国企业集团内部传输，须获得监管机构批准，审批周期较长。

![跨境数据传输机制](../../../assets/images/chapter_09/03_Cross_Border_Data_Transfer_v6.png)

### 传输影响评估（TIA）

Schrems II 判决要求，使用 SCC 等合同机制时，控制者须评估目的国法律是否能确保与欧盟"实质等同"的保护水平。传输影响评估应包含：

目的国法律分析：政府访问数据的法律框架（如美国 FISA 702、CLOUD Act、中国网络安全法、数据安全法）、数据主体在目的国的权利救济途径、独立监管机构的存在与有效性

传输情境评估：实际传输的数据类别与敏感程度、数据量级、接收方的访问范围与权限

补充措施设计：若目的国法律存在风险，需评估补充措施能否弥补保护差距：

- 技术措施：传输中与静态加密，密钥由欧盟实体控制
- 组织措施：访问权限最小化、审计日志
- 合同措施：强化 SCC 条款，如要求接收方质疑过度的政府请求
- 透明度措施：接收方发布政府访问请求年度报告

评估结论与触发条件：评估结论需明确传输是否可继续进行。同时需定义重新评估的触发条件，如目的国法律变化、相关判例、传输数据类型变更等。

常见误区：

- 仅依赖 SCC 签署而未进行 TIA，不符合 Schrems II 要求
- TIA 流于形式，未真实评估目的国法律风险
- 补充措施设计脱离实际，如声称"加密"但密钥仍存储在目的国

验证方法：

- 审查 TIA 文档的完整性，确认覆盖法律分析、传输情境、补充措施三个层面
- 对比 TIA 结论与实际传输状态的一致性
- 定期复审 TIA，特别是目的国法律环境发生变化时

## 9.7.6 供应商数据泄露响应

### 泄露通知时效要求

DPA 通常约定供应商在发现数据泄露后的一定时限内通知控制者（常见约定为 24 小时内）。这一时限设计需为控制者预留足够时间完成 GDPR 第 33 条规定的 72 小时监管通知义务。

收到供应商泄露通知后，控制者的响应流程包括：

1. 确认通知时效：核实供应商是否在约定时限内通知，作为后续合同责任评估的依据
2. 评估影响范围：确定受影响的数据主体数量、数据类别、泄露性质（未授权访问/外泄/破坏）
3. 判断通知义务：基于风险评估结果，决定是否需向监管机构与数据主体通知
4. 要求供应商补救：向供应商发出正式补救要求，包括根因分析、遏制措施、永久修复计划
5. 审查供应商关系：根据泄露严重程度，评估是否需调整与该供应商的合作关系

### 泄露影响评估

影响评估需综合以下因素：

数据量级：受影响记录数量。数量级越大，监管关注度与数据主体损害风险越高。

数据敏感度：是否涉及特殊类别数据（健康、生物识别等）。敏感数据泄露通常需通知数据主体。

泄露性质：未授权访问（数据可能被查看但未被复制）、外泄（数据被复制转移）、勒索加密（数据可用性受损）。外泄的风险通常高于仅访问。

保护措施状态：数据是否已加密且密钥未泄露。有效加密可显著降低风险等级。

### 供应商补救要求

控制者应正式要求供应商提供：

- 根因分析报告（约定时限内）
- 已采取的遏制措施
- 永久修复措施与时间表
- 影响范围的精确确认
- 第三方安全审计（针对严重事件）
- 补救措施的验证证据

同时需援引 DPA 中的违约条款，明确保留索赔权利。

运行指标：

- 供应商泄露通知时效达标率：在约定时限内通知的事件占比
- 从供应商通知到监管通知的平均耗时
- 供应商泄露事件后的合同调整率

## 本节要点

供应商隐私管理的核心在于：通过 DPA 将 GDPR 义务流转至处理者，通过风险评估实现差异化管控，通过持续审计验证承诺执行。

DPA 是法定要求：GDPR 第 28 条强制要求控制者与处理者签署书面协议，八项处理者义务须逐项约定。

风险分级驱动管控强度：基于数据访问风险、供应商成熟度、地理监管风险的综合评估，对应差异化的审批、评估、审计与复审要求。

子处理者管理须闭环：建立新增通知订阅、变更评估、反对权行使的完整流程，避免数据流向失控。

国际传输需 TIA 支撑：SCC 签署不能替代传输影响评估，需评估目的国法律风险并设计补充措施。

泄露响应需预案：供应商泄露通知时限应为监管通知预留窗口，响应流程需预先明确。
